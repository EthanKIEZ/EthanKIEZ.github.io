> **适合人群**：正在学习 OSTEP 第六章 Limited Direct Execution，想理解操作系统如何让程序高效、安全地运行在 CPU 上的读者。
{: .notice--info}

## 1. 什么是受限直接执行

让程序运行在 CPU 上最简单的方式，就是操作系统把程序加载到内存，然后让 CPU 一条一条执行它的指令。这种方式叫**直接执行**。

但如果没有限制，程序一旦运行起来就可以：

- 一直占用 CPU，操作系统无法收回
- 随意访问任何内存，包括操作系统自己的内存
- 直接操作硬件

这样操作系统就无法管理资源、无法保护系统，最终只能沦为一个“提供一些辅助函数的库”。

所以现代操作系统采用**受限直接执行（Limited Direct Execution）**：程序大部分时间直接在 CPU 上运行，但在关键时刻通过**陷阱**进入内核，由操作系统接管。

---

## 2. 用户模式与内核模式

CPU 有两种运行模式：

| | 用户模式 | 内核模式 |
|--|---------|---------|
| 运行代码 | 普通应用程序 | 操作系统内核 |
| 权限 | 低，只能访问自己的内存 | 高，可以访问所有内存和硬件 |
| 能做的事 | 普通计算、访问自己的地址空间 | 管理进程、内存、设备、文件系统 |
| 切换方式 | 通过系统调用或中断进入内核 | 处理完后返回用户模式 |

分开的原因是保护系统：普通程序不能直接破坏硬件或其他进程，必须通过受控的方式请求操作系统服务。

---

## 3. 特权操作

特权操作是指**不能只由普通程序自己决定**，必须由操作系统统一管控的操作。

例如“从磁盘读取”是特权操作，因为磁盘是共享硬件，如果每个程序都能直接读写任意位置，就可能读取他人文件或破坏文件系统。

其他常见特权操作：

| 操作 | 为什么需要特权 |
|------|---------------|
| 读写磁盘/文件系统 | 防止越权访问、保护数据完整性 |
| 访问任意物理内存 | 隔离进程地址空间 |
| 创建/销毁进程 | 防止资源被滥用 |
| 修改 CPU 运行模式 | 只有内核能进入内核模式 |
| 配置中断、时钟 | 调度与抢占由内核管理 |
| 直接访问硬件设备 | 统一管理和仲裁 |
| 修改页表/虚拟内存映射 | 内存管理是内核职责 |
| 设置网络配置 | 涉及系统级安全策略 |

---

## 4. 陷阱机制

### 4.1 陷阱指令

陷阱指令是让程序从用户模式主动进入内核模式的特殊 CPU 指令。

| 架构 | 指令 |
|------|------|
| 早期 x86 Linux | `int 0x80` |
| 现代 x86_64 | `syscall` / `sysenter` |
| ARM | `svc` |
| RISC-V | `ecall` |

触发陷阱的常见情况：

- **系统调用**：程序主动请求操作系统服务，如 `open()`、`read()`、`fork()`。
- **异常**：程序执行了非法操作，如除以零、访问非法内存。
- **中断**：硬件发出的信号，如时钟中断、键盘中断。

### 4.2 陷阱表

陷阱表是一张“地址对照表”，记录了每种陷阱对应的内核处理函数地址。

操作系统启动时初始化陷阱表，并把它在内存中的位置告诉 CPU。之后发生陷阱时，CPU 就能查表并跳转到正确的处理函数。

如果某种陷阱没有对应的表项，CPU 会跳转到无效地址，通常导致系统崩溃或内核 panic。

### 4.3 为什么进入内核时要保存寄存器

CPU 寄存器数量是有限的。当程序通过陷阱进入内核时，内核代码也会使用这些寄存器。如果不先把用户程序的寄存器值保存起来，内核就会覆盖它们。

等内核处理完，执行 `return-from-trap` 返回用户程序时，需要恢复：

- 程序计数器，知道回到哪条指令
- 栈指针，知道用户栈位置
- 通用寄存器里的参数、返回值、临时数据
- 状态寄存器

所以操作系统在进入内核时保存寄存器，退出内核时恢复寄存器，保证用户程序能从暂停的地方继续正确执行。

## 5. 系统调用为什么看起来像过程调用

系统调用看起来和普通 C 函数调用一样，是因为 C 标准库给每个系统调用包了一层包装函数。

例如调用 `open("file.txt", O_RDONLY)` 时：

1. C 库把参数放到约定位置（寄存器或栈）。
2. 把系统调用号放到约定位置。
3. 执行陷阱指令（如 `syscall`）。
4. CPU 进入内核，操作系统根据系统调用号执行真正的逻辑。
5. 内核返回结果，C 库再把结果传回程序。

所以系统调用**本质是陷阱**，但 C 库把它包成了普通函数的样子。

## 6. 创建进程时的四个步骤

操作系统创建新进程时通常要做以下四件事：

| 步骤 | 含义 |
|------|------|
| 在进程列表上创建条目 | 在进程表/PCB 表中给新进程建档，记录 PID、状态、fd 表等 |
| 为程序分配内存 | 准备地址空间：代码段、数据段、堆、栈 |
| 将程序加载到内存中 | 把可执行文件从磁盘读入内存 |
| 根据 `argc/argv` 设置程序栈 | 把命令行参数放到栈上，供 `main()` 使用 |

## 7. 受限直接运行协议概览

下表概括了 OSTEP 表 6.2 描述的受限直接运行协议。

### 7.1 启动阶段

| 操作系统 | 硬件 |
|----------|------|
| 初始化陷阱表 | 记住系统调用处理程序的地址 |

操作系统启动时设置陷阱表，让硬件知道发生陷阱时该跳转到哪里。

### 7.2 运行程序前

| 操作系统 | 硬件 | 程序 |
|----------|------|------|
| 在进程列表上创建条目 | | |
| 为程序分配内存 | | |
| 将程序加载到内存中 | | |
| 根据 `argv` 设置程序栈 | | |
| 用寄存器/程序计数器填充内核栈 | | |
| 从陷阱返回 | 从内核栈恢复寄存器；转向用户模式；跳到 `main` | |

操作系统为新进程准备好一切后，通过“从陷阱返回”切换到用户模式，程序从 `main()` 开始执行。

### 7.3 程序运行时

| 程序 | 硬件 | 操作系统 |
|------|------|----------|
| 运行 `main` | | |
| 调用系统调用 | 陷入操作系统 | |
| | 将寄存器保存到内核栈 | |
| | 转向内核模式 | |
| | 跳到陷阱处理程序 | |
| | | 处理陷阱 |
| | | 做系统调用的工作 |
| | 从陷阱返回 | |
| | 从内核栈恢复寄存器 | |
| | 转向用户模式 | |
| | 跳到陷阱之后的程序计数器 | |
| 继续运行 | | |

程序在用户模式下运行，调用系统调用时进入内核，操作系统处理完后再返回用户模式，从系统调用的下一条指令继续执行。

### 7.4 程序结束时

| 程序 | 硬件 | 操作系统 |
|------|------|----------|
| 从 `main` 返回 | | |
| 调用 `exit()` | 陷入操作系统 | |
| | | 释放进程的内存 |
| | | 将进程从进程列表中清除 |

程序通过 `exit()` 陷入内核，操作系统回收资源并销毁进程。

## 总结

受限直接执行的核心思想：

> 程序大部分时间直接在 CPU 上运行，但在系统调用、中断、异常等关键时刻通过陷阱进入内核，由操作系统接管，处理完后再返回用户模式继续执行。

理解这一点，就掌握了操作系统虚拟化 CPU 的基础机制。

## 8. yield 系统调用

`yield` 系统调用（POSIX 中对应 `sched_yield()`）让当前正在运行的进程**主动放弃 CPU**，重新回到就绪队列，让调度器挑选另一个进程执行。

```c
#include <sched.h>
int sched_yield(void);
```

调用后进程不阻塞，只是自愿让出当前这次 CPU 时间。它常用于自旋锁等待、协作式礼让等场景，避免忙等待浪费 CPU。

### 8.1 多核 CPU 上还需要 yield 吗？

现代操作系统不靠 `yield` 拿回控制权。即使没有 `yield`，操作系统也能通过**时钟中断**强制夺回 CPU：时间片用完时，时钟中断触发，调度器决定下一个运行谁。

所以 `yield` 不是必须的，它只是**合作式地**让出 CPU，让调度提前发生。

### 8.2 为什么有时钟中断还需要 yield？

时钟中断只能按固定间隔触发。一个进程可能早就没事干了，却还要等到下一次时钟中断才被发现。`yield` 允许进程在知道该让出 CPU 时主动让出，减少响应延迟和空转浪费。

两者互补：

| 机制 | 作用 |
|------|------|
| 时钟中断 | 防止进程霸占 CPU 太久，强制切换 |
| yield | 进程主动让出，避免空转和延迟 |

### 8.3 第四章模拟器为什么看起来不需要时钟中断？

OSTEP 的 `process-run.py` 等模拟器里的进程通常会主动发起 I/O 请求。发起 I/O 时触发陷阱进入操作系统，操作系统发现它需要等待 I/O，于是把它阻塞并调度其他进程。

所以模拟器里的进程“没事干”时会主动通过陷阱告诉操作系统，不需要等时钟中断。但真实系统中，纯 CPU 计算的进程可能不做 I/O 也不调用 yield，必须靠时钟中断强制切换。

## 9. 内核栈

### 9.1 为什么叫内核栈

进程从用户态陷入内核态时，寄存器和状态被保存到**内核栈**。它之所以叫内核栈，是因为：

- 它位于内核地址空间，用户程序无法访问
- 只在 CPU 处于内核态时使用
- 由操作系统分配和管理，每个进程通常有自己对应的内核栈

### 9.2 内核栈的用途

除了保存陷入时的寄存器现场，内核栈还用于：

| 用途 | 说明 |
|------|------|
| 保存上下文 | 中断/系统调用返回时恢复用户态现场 |
| 内核函数调用 | 内核处理系统调用、中断时的栈帧 |
| 中断处理 | 中断处理程序的临时数据 |
| 隔离与安全 | 用户态无法修改，防止篡改返回地址和寄存器 |
| 嵌套中断 | 中断发生时已在内核态，继续压入新现场 |

### 9.3 用户栈与内核栈对比

| | 用户栈 | 内核栈 |
|--|--------|--------|
| 使用时机 | 用户态代码运行时 | 内核态代码运行时 |
| 地址空间 | 用户地址空间 | 内核地址空间 |
| 访问权限 | 用户可读写 | 只有内核可访问 |
| 主要用途 | 局部变量、函数调用 | 保存现场、内核函数调用、中断处理 |
| 大小 | 通常较大（如 8MB） | 通常较小（如 4KB–16KB） |

## 10. 上下文切换

### 10.1 为什么切换后 B 看起来像刚陷入内核

当操作系统从进程 A 切换到进程 B 时：

1. 保存 A 的寄存器到 A 的进程结构
2. 从 B 的进程结构中恢复 B 的寄存器
3. 把当前内核栈切换成 B 的内核栈
4. 执行中断返回指令（如 `iret`）

此时 CPU 看到 B 的内核栈上保存的是 B 上一次陷入内核时的用户态现场（程序计数器、栈指针等）。中断返回后，CPU 认为自己是刚从 B 的用户态进入内核、现在要返回，于是从 B 原来的位置继续执行。

所以上下文切换本质上是在“伪造”一个历史现场，让系统看起来像是 B 刚刚陷入内核。

### 10.2 xv6 的 swtch 汇编

`swtch` 是 xv6 中实现上下文切换的汇编函数，保存旧进程的寄存器并恢复新进程的寄存器。

```asm
.globl swtch
swtch:
    movl 4(%esp), %eax      # eax = old（指向旧 context 指针的指针）
    popl 0(%eax)            # 保存返回地址到旧 context.eip
    movl %esp, 4(%eax)      # 保存栈指针到旧 context.esp
    movl %ebx, 8(%eax)      # 保存 ebx
    movl %ecx, 12(%eax)     # 保存 ecx
    movl %edx, 16(%eax)     # 保存 edx
    movl %esi, 20(%eax)     # 保存 esi
    movl %edi, 24(%eax)     # 保存 edi
    movl %ebp, 28(%eax)     # 保存 ebp

    movl 4(%esp), %eax      # eax = new（指向新 context 的指针）
    movl 28(%eax), %ebp     # 恢复 ebp
    movl 24(%eax), %edi     # 恢复 edi
    movl 20(%eax), %esi     # 恢复 esi
    movl 16(%eax), %edx     # 恢复 edx
    movl 12(%eax), %ecx     # 恢复 ecx
    movl 8(%eax), %ebx      # 恢复 ebx
    movl 4(%eax), %esp      # 恢复栈指针，切换栈
    pushl 0(%eax)           # 把新 context.eip 压入新栈
    ret                     # 弹出返回地址，跳转到新进程执行
```

关键步骤：
- 保存旧进程的 eip、esp、通用寄存器
- 恢复新进程的通用寄存器
- 切换 esp 到新进程的栈
- 用 `ret` 跳转到新进程之前暂停的位置

## 11. 禁用中断

操作系统可以在中断处理期间**禁止中断**，防止在处理一个中断时被另一个中断打断。

但禁用中断时间过长会导致**丢失中断**：

- 硬件通常只维护一个 pending 位
- 同类型中断在禁用期间多次到来，pending 位仍然是 1
- CPU 处理完一次后只看到一次中断，后续事件被吞掉

后果举例：

| 中断来源 | 丢失/延迟的后果 |
|----------|----------------|
| 网卡 | 接收缓冲区满，数据包被丢弃 |
| 磁盘控制器 | I/O 完成确认延迟，吞吐量下降 |
| 串口/UART | 接收缓冲区溢出，字符丢失 |
| 定时器 | 系统时间不准，调度变粗 |

所以禁用中断只能用于极短的关键区段，不能长时间关闭。

## 12. 测量作业

OSTEP 第六章提供了三个测量程序，用于测量系统调用、上下文切换等开销。

### 12.1 measure_gettimeofday

测量 `gettimeofday()` 的计时器分辨率。

```c
for (int i = 0; i < 1000000; i++) {
    gettimeofday(&t2, NULL);
    long delta = ...;
    if (delta > 0 && delta < min_delta) {
        min_delta = delta;
    }
    t1 = t2;
}
```

连续调用 100 万次，取相邻两次的最小非零差值。结果通常是 `1 微秒`，说明计时器最小分辨率为 1 微秒。

### 12.2 measure_syscall

测量一次系统调用的开销。

```c
for (int i = 0; i < n; i++) {
    read(0, buf, 0);
}
```

`read(0, buf, 0)` 读取 0 字节，触发进入内核但不做实际 I/O，因此接近纯系统调用开销。在 macOS 上测得约 `0.17 微秒/次`。

### 12.3 measure_ctxswitch

测量进程上下文切换开销。

```c
pipe(p1);
pipe(p2);
fork();

// 父进程
for (int i = 0; i < n; i++) {
    write(p1[1], &c, 1);
    read(p2[0], &c, 1);
}

// 子进程
for (int i = 0; i < n; i++) {
    read(p1[0], &c, 1);
    write(p2[1], &c, 1);
}
```

父子进程通过两个管道 ping-pong 传一个字符。一次往返触发两次上下文切换，因此：

```c
ctxswitch = total / n / 2;
```

在 macOS 上测得约 `2.9 微秒/次`。

### 12.4 为什么需要两个管道

一个管道只能单向传输，而且管道有缓冲区。如果只用单向管道，父进程可能一口气写满缓冲区，子进程再一口气读空，只触发 2 次上下文切换，无法测出每次切换。

两个管道形成 ping-pong 环路，强制父子进程交替阻塞和唤醒，每轮稳定产生 2 次上下文切换。

### 12.5 为什么迭代次数越大，每次系统调用越小

- **固定开销被摊薄**：启动、`gettimeofday` 等一次性开销被更多次迭代平均
- **CPU 缓存热身**：代码、数据、TLB 命中率提高
- **CPU 频率提升**：持续负载让 CPU 进入高性能状态
- **分支预测稳定**：循环分支被预测器学会
- **统计噪声被稀释**：偶尔的中断、调度波动对大样本影响更小

## 13. gettimeofday、vDSO 与时间戳

### 13.1 gettimeofday 的输入输出

```c
int gettimeofday(struct timeval *tv, struct timezone *tz);
```

- `tv`：指向 `timeval` 结构体的指针，函数把当前时间填到这里
- `tz`：已废弃，通常传 `NULL`

```c
struct timeval {
    time_t      tv_sec;   // 秒
    suseconds_t tv_usec;  // 微秒
};
```

返回值：成功 0，失败 -1 并设置 errno。

### 13.2 vDSO 时间戳与 syscall 的区别

| 特性 | syscall 方式 | vDSO 方式 |
|------|-------------|-----------|
| 是否进入内核 | 是 | 否 |
| 特权级切换 | 需要 | 不需要 |
| 性能 | 较慢（几百纳秒级） | 很快（几十纳秒级） |
| 适用操作 | 所有系统调用 | 仅限只读、简单操作（如取时间） |

`gettimeofday()` 在 Linux 上通常通过 vDSO 实现，不经过真正的系统调用。

### 13.3 时间戳

时间戳（Timestamp）是用一个数值表示某个特定时刻，最常见的是 Unix 时间戳：从 1970-01-01 00:00:00 UTC 开始经过的秒数（或毫秒、微秒、纳秒）。它便于计算机存储和计算，常用于日志、数据库、缓存过期等场景。

## 14. EIP 与 ESP

### 14.1 EIP（Extended Instruction Pointer）

**扩展指令指针寄存器**，存放下一条要执行的指令在内存中的地址。

```text
EIP = 0x08048400  →  CPU 接下来会执行地址 0x08048400 处的指令
```

程序运行时，CPU 不断做：

```text
取 EIP 指向的指令 → 执行 → EIP 增加 → 取下一条
```

当发生函数调用、跳转、中断或上下文切换时，`EIP` 会被修改或保存。

### 14.2 ESP（Extended Stack Pointer）

**扩展栈指针寄存器**，指向当前栈的栈顶位置。

```text
ESP = 0x7fff1000  →  栈顶在地址 0x7fff1000
```

栈是从高地址向低地址增长的：

```text
高地址
┌─────────┐
│  数据   │
├─────────┤
│  数据   │
├─────────┤ ← ESP 指向这里（栈顶）
│         │
低地址
```

`push` 指令把数据压栈，`ESP` 减小；`pop` 指令把数据弹出，`ESP` 增大。

### 14.3 在上下文切换中的作用

在 xv6 的 `swtch` 汇编中：

```asm
popl 0(%eax)        # 保存返回地址到旧 context.eip
movl %esp, 4(%eax)  # 保存栈指针到旧 context.esp
```

- `EIP` 保存的是进程暂停时下一条要执行的指令地址，恢复后进程从那里继续。
- `ESP` 保存的是进程当时的栈顶位置，恢复后栈就回到正确的位置。

### 14.4 一句话总结

> **EIP 告诉 CPU 下一条执行哪条指令，ESP 告诉 CPU 栈顶在哪里。** 它们是进程现场中最关键的两个寄存器。
