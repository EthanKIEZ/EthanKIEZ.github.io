## 1. 什么是受限直接执行

让程序运行在 CPU 上最简单的方式，就是操作系统把程序加载到内存，然后让 CPU 一条一条执行它的指令。这种方式叫**直接执行**。

但如果没有限制，程序一旦运行起来就可以：

- 一直占用 CPU，操作系统无法收回
- 随意访问任何内存，包括操作系统自己的内存
- 直接操作硬件

这样操作系统就无法管理资源、无法保护系统，最终只能沦为一个“提供一些辅助函数的库”。

所以现代操作系统采用**受限直接执行（Limited Direct Execution）**：程序大部分时间直接在 CPU 上运行，但在关键时刻通过**陷阱**进入内核，由操作系统接管。

---

## 2. 用户模式与内核模式

CPU 有两种运行模式：

| | 用户模式 | 内核模式 |
|--|---------|---------|
| 运行代码 | 普通应用程序 | 操作系统内核 |
| 权限 | 低，只能访问自己的内存 | 高，可以访问所有内存和硬件 |
| 能做的事 | 普通计算、访问自己的地址空间 | 管理进程、内存、设备、文件系统 |
| 切换方式 | 通过系统调用或中断进入内核 | 处理完后返回用户模式 |

分开的原因是保护系统：普通程序不能直接破坏硬件或其他进程，必须通过受控的方式请求操作系统服务。

---

## 3. 特权操作

特权操作是指**不能只由普通程序自己决定**，必须由操作系统统一管控的操作。

例如“从磁盘读取”是特权操作，因为磁盘是共享硬件，如果每个程序都能直接读写任意位置，就可能读取他人文件或破坏文件系统。

其他常见特权操作：

| 操作 | 为什么需要特权 |
|------|---------------|
| 读写磁盘/文件系统 | 防止越权访问、保护数据完整性 |
| 访问任意物理内存 | 隔离进程地址空间 |
| 创建/销毁进程 | 防止资源被滥用 |
| 修改 CPU 运行模式 | 只有内核能进入内核模式 |
| 配置中断、时钟 | 调度与抢占由内核管理 |
| 直接访问硬件设备 | 统一管理和仲裁 |
| 修改页表/虚拟内存映射 | 内存管理是内核职责 |
| 设置网络配置 | 涉及系统级安全策略 |

## 4. 陷阱机制

### 4.1 陷阱指令

陷阱指令是让程序从用户模式主动进入内核模式的特殊 CPU 指令。

| 架构 | 指令 |
|------|------|
| 早期 x86 Linux | `int 0x80` |
| 现代 x86_64 | `syscall` / `sysenter` |
| ARM | `svc` |
| RISC-V | `ecall` |

触发陷阱的常见情况：

- **系统调用**：程序主动请求操作系统服务，如 `open()`、`read()`、`fork()`。
- **异常**：程序执行了非法操作，如除以零、访问非法内存。
- **中断**：硬件发出的信号，如时钟中断、键盘中断。

### 4.2 陷阱表

陷阱表是一张“地址对照表”，记录了每种陷阱对应的内核处理函数地址。

操作系统启动时初始化陷阱表，并把它在内存中的位置告诉 CPU。之后发生陷阱时，CPU 就能查表并跳转到正确的处理函数。

如果某种陷阱没有对应的表项，CPU 会跳转到无效地址，通常导致系统崩溃或内核 panic。

### 4.3 为什么进入内核时要保存寄存器

CPU 寄存器数量是有限的。当程序通过陷阱进入内核时，内核代码也会使用这些寄存器。如果不先把用户程序的寄存器值保存起来，内核就会覆盖它们。

等内核处理完，执行 `return-from-trap` 返回用户程序时，需要恢复：

- 程序计数器，知道回到哪条指令
- 栈指针，知道用户栈位置
- 通用寄存器里的参数、返回值、临时数据
- 状态寄存器

所以操作系统在进入内核时保存寄存器，退出内核时恢复寄存器，保证用户程序能从暂停的地方继续正确执行。

## 5. 系统调用为什么看起来像过程调用

系统调用看起来和普通 C 函数调用一样，是因为 C 标准库给每个系统调用包了一层包装函数。

例如调用 `open("file.txt", O_RDONLY)` 时：

1. C 库把参数放到约定位置（寄存器或栈）。
2. 把系统调用号放到约定位置。
3. 执行陷阱指令（如 `syscall`）。
4. CPU 进入内核，操作系统根据系统调用号执行真正的逻辑。
5. 内核返回结果，C 库再把结果传回程序。

所以系统调用**本质是陷阱**，但 C 库把它包成了普通函数的样子。

## 6. 创建进程时的四个步骤

操作系统创建新进程时通常要做以下四件事：

| 步骤 | 含义 |
|------|------|
| 在进程列表上创建条目 | 在进程表/PCB 表中给新进程建档，记录 PID、状态、fd 表等 |
| 为程序分配内存 | 准备地址空间：代码段、数据段、堆、栈 |
| 将程序加载到内存中 | 把可执行文件从磁盘读入内存 |
| 根据 `argc/argv` 设置程序栈 | 把命令行参数放到栈上，供 `main()` 使用 |

## 7. 受限直接运行协议概览

下表概括了 OSTEP 表 6.2 描述的受限直接运行协议。

### 7.1 启动阶段

| 操作系统 | 硬件 |
|----------|------|
| 初始化陷阱表 | 记住系统调用处理程序的地址 |

操作系统启动时设置陷阱表，让硬件知道发生陷阱时该跳转到哪里。

### 7.2 运行程序前

| 操作系统 | 硬件 | 程序 |
|----------|------|------|
| 在进程列表上创建条目 | | |
| 为程序分配内存 | | |
| 将程序加载到内存中 | | |
| 根据 `argv` 设置程序栈 | | |
| 用寄存器/程序计数器填充内核栈 | | |
| 从陷阱返回 | 从内核栈恢复寄存器；转向用户模式；跳到 `main` | |

操作系统为新进程准备好一切后，通过“从陷阱返回”切换到用户模式，程序从 `main()` 开始执行。

### 7.3 程序运行时

| 程序 | 硬件 | 操作系统 |
|------|------|----------|
| 运行 `main` | | |
| 调用系统调用 | 陷入操作系统 | |
| | 将寄存器保存到内核栈 | |
| | 转向内核模式 | |
| | 跳到陷阱处理程序 | |
| | | 处理陷阱 |
| | | 做系统调用的工作 |
| | 从陷阱返回 | |
| | 从内核栈恢复寄存器 | |
| | 转向用户模式 | |
| | 跳到陷阱之后的程序计数器 | |
| 继续运行 | | |

程序在用户模式下运行，调用系统调用时进入内核，操作系统处理完后再返回用户模式，从系统调用的下一条指令继续执行。

### 7.4 程序结束时

| 程序 | 硬件 | 操作系统 |
|------|------|----------|
| 从 `main` 返回 | | |
| 调用 `exit()` | 陷入操作系统 | |
| | | 释放进程的内存 |
| | | 将进程从进程列表中清除 |

程序通过 `exit()` 陷入内核，操作系统回收资源并销毁进程。

## 总结

受限直接执行的核心思想：

> 程序大部分时间直接在 CPU 上运行，但在系统调用、中断、异常等关键时刻通过陷阱进入内核，由操作系统接管，处理完后再返回用户模式继续执行。

理解这一点，就掌握了操作系统虚拟化 CPU 的基础机制。
