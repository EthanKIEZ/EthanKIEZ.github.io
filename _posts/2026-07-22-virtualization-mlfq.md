## 1. 为什么要区分 CPU 密集型和 I/O 密集型

### 两类进程的特点

| 类型 | 行为 | 例子 |
|---|---|---|
| **CPU 密集型** | 长时间占用 CPU 计算，很少 I/O | 视频编码、科学计算、大数据处理 |
| **I/O 密集型** | 频繁发起 I/O，CPU 运行时间短 | 终端输入、编辑器、浏览器、数据库查询 |

### 为什么要区分

#### 1. 响应时间需求不同

I/O 密集型进程通常直接面对用户：

```text
你按下一个键 → 编辑器应该 10ms 内显示字符
你点击一个链接 → 浏览器应该尽快开始加载
```

如果让 CPU 密集型进程一直占着 CPU，你的按键可能要等几秒才响应。

CPU 密集型进程则不在乎那几十毫秒，它需要的是长时间稳定运行。

#### 2. I/O 密集型进程会让出 CPU

I/O 密集型进程运行一会儿就会去等 I/O：

```text
编辑器运行 1ms → 读键盘输入 → 阻塞等待
浏览器运行 2ms → 发网络请求 → 阻塞等待
```

它们主动让出 CPU，不会霸占。所以给它们高优先级是"安全的"——它们用一会儿 CPU 就自己去睡觉了。

#### 3. CPU 密集型进程会霸占 CPU

CPU 密集型进程一旦上 CPU，就会一直跑：

```text
视频编码：跑 10 分钟，几乎不做 I/O
```

如果不加控制，它会让其他所有进程都饿死。

#### 4. 提高系统整体效率

理想情况：

```text
I/O 密集型进程：高优先级，快速响应，发起 I/O 后阻塞
                ↓
CPU 密集型进程：在 I/O 等待期间运行，不浪费 CPU
```

这样 CPU 和 I/O 设备都能保持忙碌，系统吞吐量最高。

#### 5. 避免饿死

如果不区分类型，纯按 FIFO 或简单优先级调度：

```text
CPU 密集型进程一直跑
I/O 密集型进程永远排不上
```

用户会感觉系统"卡死"了。

### 一句话总结

> 区分 CPU 密集型和 I/O 密集型，是为了让需要快速响应的进程优先运行，让需要长时间计算的进程利用间隙运行，从而提高响应速度和整体资源利用率。
