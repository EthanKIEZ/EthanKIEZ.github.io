## 1. 调度指标：周转时间

衡量调度算法性能的一个重要指标是**周转时间（Turnaround Time）**。

### 公式

```text
周转时间 = 完成时间 − 到达时间
```

### 定义

| 概念 | 定义 |
|---|---|
| 到达时间 | 作业或进程到达系统、进入就绪队列的时刻 |
| 完成时间 | 作业或进程完成执行、离开系统的时刻 |
| 周转时间 | 从到达系统到完成所经历的总时间 |

---

## 2. FIFO 调度示例

3 个工作 A、B、C 几乎同时到达（到达时间都为 0），FIFO 调度，每个工作运行 10s。

```text
A: 0 ~ 10s
B: 10 ~ 20s
C: 20 ~ 30s
```

完成时间：A=10，B=20，C=30。

平均周转时间：

```text
(10 + 20 + 30) / 3 = 20s
```

## 3. FIFO 与 SJF 对比

当工作长度不同时，不同调度算法的平均周转时间差异很大。

假设 3 个工作同时到达：

| 工作 | 运行时间 |
|---|---|
| A | 100s |
| B | 10s |
| C | 10s |

### FIFO 调度

```text
A: 0 ~ 100s
B: 100 ~ 110s
C: 110 ~ 120s
```

平均周转时间：

```text
(100 + 110 + 120) / 3 = 110s
```

### SJF 调度

SJF 先运行最短的工作：

```text
B: 0 ~ 10s
C: 10 ~ 20s
A: 20 ~ 120s
```

平均周转时间：

```text
(10 + 20 + 120) / 3 = 50s
```

当工作长度差异大时，SJF 的平均周转时间明显优于 FIFO。

## 4. process-run.py 模拟器中的时间计算

### io_isdone 的含义

在 `process-run.py` 中，进程执行 `io` 指令后会进入 `BLOCKED` 状态，等待 I/O 完成。`io_isdone` 表示 I/O 完成的事件，此时进程从 `BLOCKED` 变为 `READY`。

I/O 完成事件本身不消耗 CPU 周期，它只是状态转换的通知。

### 三个进程的周转时间计算

三个进程同时到达，使用 FIFO 调度，I/O 长度为 1：

```text
P1: cpu1 io1
P2: io1 cpu1
P3: cpu1 io1 cpu1
```

在 `process-run.py` 内部，`io` 指令会展开为 `io` 和 `io_done` 两条指令，每条各占 1 个 CPU 周期。

实际运行时间线：

```text
t=1: P1 运行 cpu
t=2: P1 运行 io，发起 I/O，P1 阻塞
     P2 运行 io，发起 I/O，P2 阻塞
     P3 运行 cpu
t=3: P3 运行 io，发起 I/O，P3 阻塞
     P1 I/O 完成，P1 就绪
t=4: P1 运行 io_done，P1 完成
     P2 I/O 完成，P2 就绪
t=5: P2 运行 io_done
t=6: P3 I/O 完成，P3 就绪
     P2 运行 cpu，P2 完成
t=7: P3 运行 io_done
t=8: P3 运行 cpu，P3 完成
```

完成时间：P1=4，P2=6，P3=8。

平均周转时间：

```text
(4 + 6 + 8) / 3 = 6
```

## 5. 一句话总结

> 周转时间衡量一个作业在系统中停留多久。FIFO 简单但可能让短作业等待过长；SJF 优先运行短作业，能降低平均周转时间。在 process-run.py 中，`io` 和 `io_done` 各占一个 CPU 周期，I/O 完成事件本身不占 CPU。
