## 1. 调度指标：周转时间

衡量调度算法性能的一个重要指标是**周转时间（Turnaround Time）**。

### 公式

```text
周转时间 = 完成时间 − 到达时间
```

### 定义

| 概念 | 定义 |
|---|---|
| 到达时间 | 作业或进程到达系统、进入就绪队列的时刻 |
| 完成时间 | 作业或进程完成执行、离开系统的时刻 |
| 周转时间 | 从到达系统到完成所经历的总时间 |

---

## 2. FIFO 调度示例

3 个工作 A、B、C 几乎同时到达（到达时间都为 0），FIFO 调度，每个工作运行 10s。

```text
A: 0 ~ 10s
B: 10 ~ 20s
C: 20 ~ 30s
```

完成时间：A=10，B=20，C=30。

平均周转时间：

```text
(10 + 20 + 30) / 3 = 20s
```

---

## 3. FIFO 与 SJF 对比

当工作长度不同时，不同调度算法的平均周转时间差异很大。

假设 3 个工作同时到达：

| 工作 | 运行时间 |
|---|---|
| A | 100s |
| B | 10s |
| C | 10s |

### FIFO 调度

```text
A: 0 ~ 100s
B: 100 ~ 110s
C: 110 ~ 120s
```

平均周转时间：

```text
(100 + 110 + 120) / 3 = 110s
```

### SJF 调度

SJF 先运行最短的工作：

```text
B: 0 ~ 10s
C: 10 ~ 20s
A: 20 ~ 120s
```

平均周转时间：

```text
(10 + 20 + 120) / 3 = 50s
```

当工作长度差异大时，SJF 的平均周转时间明显优于 FIFO。

## 4. process-run.py 模拟器中的时间计算

### io_isdone 的含义

在 `process-run.py` 中，进程执行 `io` 指令后会进入 `BLOCKED` 状态，等待 I/O 完成。`io_isdone` 表示 I/O 完成的事件，此时进程从 `BLOCKED` 变为 `READY`。

I/O 完成事件本身不消耗 CPU 周期，它只是状态转换的通知。

### 三个进程的周转时间计算

三个进程同时到达，使用 FIFO 调度，I/O 长度为 1：

```text
P1: cpu1 io1
P2: io1 cpu1
P3: cpu1 io1 cpu1
```

在 `process-run.py` 内部，`io` 指令会展开为 `io` 和 `io_done` 两条指令，每条各占 1 个 CPU 周期。

实际运行时间线：

```text
t=1: P1 运行 cpu
t=2: P1 运行 io，发起 I/O，P1 阻塞
     P2 运行 io，发起 I/O，P2 阻塞
     P3 运行 cpu
t=3: P3 运行 io，发起 I/O，P3 阻塞
     P1 I/O 完成，P1 就绪
t=4: P1 运行 io_done，P1 完成
     P2 I/O 完成，P2 就绪
t=5: P2 运行 io_done
t=6: P3 I/O 完成，P3 就绪
     P2 运行 cpu，P2 完成
t=7: P3 运行 io_done
t=8: P3 运行 cpu，P3 完成
```

完成时间：P1=4，P2=6，P3=8。

平均周转时间：

```text
(4 + 6 + 8) / 3 = 6
```

---

## 5. 一句话总结

> 周转时间衡量一个作业在系统中停留多久。FIFO 简单但可能让短作业等待过长；SJF 优先运行短作业，能降低平均周转时间。在 process-run.py 中，`io` 和 `io_done` 各占一个 CPU 周期，I/O 完成事件本身不占 CPU。

## 6. argc 与 argv

### 为什么叫 argc 和 argv

```c
int main(int argc, char *argv[])
```

`argc` 和 `argv` 不是 C 语言关键字，只是约定俗成的命名：

| 名字 | 来源 | 含义 |
|---|---|---|
| `argc` | **arg**ument **c**ount | 参数个数 |
| `argv` | **arg**ument **v**ector | 参数数组 |

可以改成其他名字，程序照样编译：

```c
int main(int count, char **args)
```

但全世界都用 `argc`/`argv`，所以成了默认约定。

### 操作系统如何传参

程序启动时，操作系统把命令行参数整理好传给 `main`：

```bash
./a.out hello world
```

构造：

```text
argc = 3
argv[0] = "./a.out"
argv[1] = "hello"
argv[2] = "world"
```

### 还有其他默认参数吗

标准 C 只支持两种 `main` 形式：

```c
int main(void);
int main(int argc, char *argv[]);
```

Linux 等系统常见扩展支持第三种形式：

```c
int main(int argc, char *argv[], char *envp[])
```

`envp` 是环境变量数组。

## 7. OSTEP 第七章习题解答

假设所有作业同时到达（到达时间 = 0），FIFO 按 A→B→C 顺序调度。

### 第 1 题：3 个长度为 200 的作业

#### FIFO

```text
A: 0 ~ 200
B: 200 ~ 400
C: 400 ~ 600
```

| 作业 | 响应时间 | 周转时间 |
|---|---|---|
| A | 0 | 200 |
| B | 200 | 400 |
| C | 400 | 600 |

平均响应时间 = `(0 + 200 + 400) / 3 = 200`  
平均周转时间 = `(200 + 400 + 600) / 3 = 400`

#### SJF

3 个作业长度相同，SJF 结果与 FIFO 相同。

平均响应时间 = `200`  
平均周转时间 = `400`

### 第 2 题：长度分别为 100、200、300

A=100, B=200, C=300。

#### FIFO（A→B→C）

```text
A: 0 ~ 100
B: 100 ~ 300
C: 300 ~ 600
```

平均响应时间 = `(0 + 100 + 300) / 3 ≈ 133.33`  
平均周转时间 = `(100 + 300 + 600) / 3 ≈ 333.33`

#### SJF

按长度递增 A→B→C，结果与 FIFO 相同。

### 第 3 题：RR 时间片为 1

3 个作业长度 100、200、300，RR 轮转：

```text
A B C A B C A B C ...
```

- A 在 t=300 完成
- B 在 t=500 完成
- C 在 t=600 完成

| 作业 | 响应时间 | 周转时间 |
|---|---|---|
| A | 0 | 300 |
| B | 1 | 500 |
| C | 2 | 600 |

平均响应时间 = `(0 + 1 + 2) / 3 = 1`  
平均周转时间 = `(300 + 500 + 600) / 3 ≈ 466.67`

### 第 4 题：SJF 与 FIFO 周转时间相同的工作负载

当所有作业长度相同，或 FIFO 的到达顺序已经按长度非递减排列时，SJF 与 FIFO 周转时间相同。

### 第 5 题：SJF 与 RR 响应时间相同的工作负载和量子长度

当所有作业长度相同且等于 RR 的量子长度时，两者响应时间相同。

### 第 6 题：工作长度增加时 SJF 的响应时间

SJF 中长作业的响应时间取决于前面短作业的数量和长度，而不是它自身的长度。

模拟结果（5 个等长作业）：

| 作业长度 | 各作业响应时间 | 平均响应时间 |
|---|---|---|
| 10 | [0, 10, 20, 30, 40] | 20 |
| 20 | [0, 20, 40, 60, 80] | 40 |
| 50 | [0, 50, 100, 150, 200] | 100 |
| 100 | [0, 100, 200, 300, 400] | 200 |

趋势：所有作业等长时，平均响应时间随长度线性增长。

### 第 7 题：量子长度增加时 RR 的响应时间

RR 中第 i 个作业的响应时间为：

```text
响应时间_i = i × 量子
```

随着量子长度增加，响应时间线性增加。

对于 N 个同时到达的作业，最坏情况响应时间为：

```text
最坏情况响应时间 = (N - 1) × 量子
```

平均响应时间为：

```text
平均响应时间 = (N - 1) × 量子 / 2
```

## 8. 为什么要区分 CPU 密集型和 I/O 密集型

### 两类进程的特点

| 类型 | 行为 | 例子 |
|---|---|---|
| **CPU 密集型** | 长时间占用 CPU 计算，很少 I/O | 视频编码、科学计算、大数据处理 |
| **I/O 密集型** | 频繁发起 I/O，CPU 运行时间短 | 终端输入、编辑器、浏览器、数据库查询 |

### 为什么要区分

#### 1. 响应时间需求不同

I/O 密集型进程通常直接面对用户：

```text
你按下一个键 → 编辑器应该 10ms 内显示字符
你点击一个链接 → 浏览器应该尽快开始加载
```

如果让 CPU 密集型进程一直占着 CPU，你的按键可能要等几秒才响应。

CPU 密集型进程则不在乎那几十毫秒，它需要的是长时间稳定运行。

#### 2. I/O 密集型进程会让出 CPU

I/O 密集型进程运行一会儿就会去等 I/O：

```text
编辑器运行 1ms → 读键盘输入 → 阻塞等待
浏览器运行 2ms → 发网络请求 → 阻塞等待
```

它们主动让出 CPU，不会霸占。所以给它们高优先级是"安全的"——它们用一会儿 CPU 就自己去睡觉了。

#### 3. CPU 密集型进程会霸占 CPU

CPU 密集型进程一旦上 CPU，就会一直跑：

```text
视频编码：跑 10 分钟，几乎不做 I/O
```

如果不加控制，它会让其他所有进程都饿死。

#### 4. 提高系统整体效率

理想情况：

```text
I/O 密集型进程：高优先级，快速响应，发起 I/O 后阻塞
                ↓
CPU 密集型进程：在 I/O 等待期间运行，不浪费 CPU
```

这样 CPU 和 I/O 设备都能保持忙碌，系统吞吐量最高。

#### 5. 避免饿死

如果不区分类型，纯按 FIFO 或简单优先级调度：

```text
CPU 密集型进程一直跑
I/O 密集型进程永远排不上
```

用户会感觉系统"卡死"了。

### 一句话总结

> 区分 CPU 密集型和 I/O 密集型，是为了让需要快速响应的进程优先运行，让需要长时间计算的进程利用间隙运行，从而提高响应速度和整体资源利用率。
